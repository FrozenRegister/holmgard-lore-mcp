### Feature — World-scope the Map Editor's hex/landmark tables (#319, Phase 1 of #308)
- `hexes` gains `world_id` (FK to `worlds`, nullable), `biome` (registry-validated against the same per-world `biomes` registry `room_nodes` uses, when set), and `elevation`/`moisture`/`temperature` columns. `landmarks` gains `world_id`, `region_id`, `population`, and `zone_type`/`zone_shape`/`predator_ref`/`threat_level`/`dominance_rank` — folding in what the RPG engine's `structures` table carries today.
- All new columns are additive and nullable/defaulted — no behavior change for the existing editor sync path (`/admin/map/push-hexes`, `/admin/map/push-landmarks`, `/internal/map-readback` all keep their exact current response shape).
- Removed the ad-hoc `POST /admin/map/setup-db` route and its duplicated `MAP_SCHEMA_DDL` — `hexes`/`landmarks` are created by `schema/migrations/0001_initial.sql` and are now fully governed by the normal D1 migration pipeline (migration `0019_map_tables_world_scoping.sql`), not re-run ad-hoc at request time.
- This is Phase 1 of a 4-phase unification of the map editor's hex-grid tables with the RPG engine's square-grid world-simulation tables (`worlds`/`tiles`/`structures`/`biomes`) — see #319–#322 and #308. No handler behavior changes in this phase; `world_map.ts`, `encounter-manage.ts`, and `biome-manage.ts` still read/write `tiles`/`structures` until Phase 2 (#320).
