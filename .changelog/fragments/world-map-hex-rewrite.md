### Feature — Rewrite `world_map` for hex-axial `hexes`/`landmarks` + D1 zone-type registry (#320, Phase 2 of #308)
- `world_map`'s 12 actions now read/write the hex-axial `hexes`/`landmarks` tables (shared with `holmgard-lore-editor`, world-scoped by migration `0019`/#319) instead of the retired square-grid `tiles`/`structures` tables. Coordinates are `q`/`r` (hex-axial), not `x`/`y`; zone-circle/ring membership uses hex distance (not Euclidean); `render_svg` renders true pointy-top hex polygons via axial-to-pixel conversion instead of square tiles.
- Fixed the two hidden couplings identified while planning this migration: `encounter-manage.ts`'s biome lookup and `biome-manage.ts`'s delete-guard now query `hexes` instead of the retired `tiles` table.
- Removed the hardcoded `LEGACY_BIOME_GLYPHS`/`LEGACY_BIOME_COLORS` fallback maps and the hardcoded 4-entry `ZONE_GLYPHS` map from `world-map.ts`. Biome glyph/color rendering now relies solely on the existing per-world `biomes` registry (falling back to `'?'`/`'#888888'` when nothing is registered); zone overlay glyphs in `preview` now come from a brand-new D1-backed per-world `zone_types` registry (migration `0020_zone_types.sql`, new `zone_type_manage` sub-tool mirroring `biome_manage`'s exact register/list/get/update/delete/validate/seed_defaults pattern), auto-seeded with the same 5 defaults (`perimeter`, `exclusion`, `hazard`, `territory`, `broadcast`) on every new world via `world_manage.create`/`generate`.
- `ResolvedZone.structureId` renamed to `landmarkId`; `world_map`'s `suggest_poi`/`update_poi` responses likewise return `landmarkId` (the `structureId` input parameter name is kept unchanged for backward compatibility).
