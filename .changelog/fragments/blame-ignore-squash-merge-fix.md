### Fixed a stale `.git-blame-ignore-revs` SHA (again) and made the failure mode non-fatal

- Root cause this time was more fundamental than the earlier rebase-SHA fix: this repo squash-merges every PR into one new commit on `main`, so the individual reformat commit `.git-blame-ignore-revs` referenced was orphaned the moment #493 merged — not by a rebase, but by squash-merge itself. Every PR's `Coverage` job started failing immediately, regardless of that PR's own content, because the previous hard `process.exit(1)` on a stale SHA also silently broke `patch-coverage-report.json`'s "written unconditionally" guarantee.
- Updated the entry to #493's actual squash-merge commit SHA, and documented the squash-merge implication directly in `.git-blame-ignore-revs`'s header for the next time this comes up.
- Changed `scripts/check-patch-coverage.mjs` to degrade gracefully instead of hard-failing: an unresolvable SHA is now dropped from the active ignore set (with a loud warning in both console output and the JSON report's new `warnings` field) rather than aborting the whole script before it can write anything. Worst case, patch-coverage becomes stricter for reformatted regions specifically — it no longer blocks every unrelated PR.
