### Added

- `aura_manage`: `check_save` (concentration saving throw when a concentrating character takes damage, DC = max(10, half damage)) and `check_duration` (auto-break concentration once `max_duration` elapses) — completes the concentration mechanic that was previously only half-implemented (`concentrate`/`break_concentration` existed with no way to actually break concentration from damage or time).
- `character_manage`: `search` (by name substring), `cast_spell` (validates the spell is known/prepared/a cantrip before casting — "anti-hallucination" enforcement — then consumes a `spell_slots`/`pact_magic_slots` slot and sets up concentration if required); `update` now also accepts `conditions`, `resistances`, `vulnerabilities`, `immunities`, `spellSlots`, `pactMagicSlots`, `knownSpells`, `preparedSpells`, `cantripsKnown`, `maxSpellLevel`, `concentratingOn`, `legendaryActions(Remaining)`, `legendaryResistances(Remaining)`, `hasLairActions`, `resourcePools`, and `currency` — wiring up `characters` table columns that existed in the schema but had no handler ever reading/writing them.
- New `rest_manage` tool (`sub: 'rest'`): `long_rest` (full HP, spell slots, pact magic, legendary counters, conditions, and death saves reset) and `short_rest` (pact magic only, plus an optional `healAmount`); both accept `characterId` or `partyId` to rest an entire party.
- New `scroll_manage` tool (`sub: 'scroll'`): `create`, `use` (consumes a charge), `identify`, `get_dc`, `get_details` — scrolls are `items` rows with `type: 'scroll'`, no schema migration needed.
- New `event_manage` tool (`sub: 'event'`): `emit`/`poll`/`ack`/`list_types` — a poll-based analogue of upstream's `subscribe_to_events`/`unsubscribe_from_events`, built on the previously-unused `event_inbox` table (Workers has no long-lived connection to push real pub/sub over).
- `combat_map`: `get_terrain`/`set_terrain` and `calculate_aoe` (circle/square/line shapes).

All of the above (except `event_manage`/`scroll_manage`, which reuse existing tables generically) operate on `characters` table columns that already existed in the schema — no migration needed. Previously declined as D&D-specific/out-of-scope in issue #74; implemented per the gap-analysis in #206.
