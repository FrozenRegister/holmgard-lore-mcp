### Feature — `spatial_manage`/`room_nodes` integrated with the dynamic biome registry (#290, #274 follow-up)
- `room_nodes.biome_context` is no longer a hardcoded 8-value DB `CHECK` constraint — SQLite can't drop a `CHECK` in place, so migration `0015_room_nodes_biome_registry.sql` rebuilds the table (copy rows, drop, rename) to remove it and add a `world_id` column.
- `spatial_manage`'s `generate`/`update` actions now validate `biome` against the per-world dynamic registry (`biome-manage.ts`'s `getBiomeRegistry`), using the exact same "unrestricted when a world has zero registered biomes" backward-compatible pattern already used by `world_map.ts`'s `patch`/`batch` actions. Validation only runs when a `worldId` is given.
- `generate` now accepts and stores `worldId`; `update` can reassign it; `list` gains a `worldIdFilter`; `look` reports the room's `worldId`.
- New discovery, documented in `docs/issues/HIGH-fk-enforcement-is-active-for-create-table-references.md`: FK enforcement **is** active in D1/miniflare for `REFERENCES` clauses declared inline in a `CREATE TABLE` statement (confirmed empirically, and cross-referenced against the pre-existing, still-open `combat_manage` FK bug) — this corrects a wrong assumption ("not active anywhere in this schema") left in migrations 0008/0009's comments, which are left as-is per this repo's migration-history rule.
- Added `src/__tests__/spatial-manage.test.ts` (previously zero coverage) and `tests/live/spatial-biome-registry.test.ts` (scoped to the new surface).
