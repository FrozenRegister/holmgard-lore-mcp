### Fix — `character_manage` now accepts snake_case `world_id` (#268)
- A prior fix already added `world_id = ?` WHERE-clause filtering to `list`/`search` when the caller passes camelCase `worldId`, but Zod silently drops unrecognized keys — so the issue's own reproduction (and any caller using `world_id`, the convention `time_manage`/`timeline_manage` already use) was accepted without erroring, yet never actually filtered anything, returning all 38 characters across every world instead of just the target world's ~16.
- `character_manage`'s `InputSchema` now also accepts `world_id`, normalized to `worldId` immediately after parsing (camelCase wins if both are somehow given) — covers `create`/`get`/`update`/`list`/`search`, every action that reads `a.worldId`, in one change.
- `check_continuity`'s cross-world noise (also flagged in #268) already has a working mechanism for this — the `world` param added in #259 (a freeform KV `**World:**` field match, documented in `checkContinuitySchema` in `src/tools/meta.ts`) — this looks like a caller-omission, not a code defect, so it's left alone here. `lore_manage.list`/`search` KV-level world scoping was only a "consider" in the issue, not a required fix; left as a follow-up.
